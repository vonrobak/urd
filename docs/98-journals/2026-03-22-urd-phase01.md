## Original Plan vs. What Was Built

Phase 1 in `docs/PLAN.md` called for a direct port of the bash script's scheduling model — daily/weekly/monthly cron-like schedules, fixed retention counts, and tier-based subvolume grouping. During implementation, we reconsidered the project's ambition: Urd is a **Time Machine for BTRFS on Linux**, not a Rust rewrite of the bash script. This led to a fundamental redesign of the scheduling and retention models before any code was written.

### What Changed from the Plan

| PLAN.md specified | What was built instead | Why |
|---|---|---|
| `Schedule` enum: Daily, Weekly, Monthly, FirstSaturday, Never | `Interval` type: `"15m"`, `"1h"`, `"1d"`, `"1w"` | Time Machine backs up frequently, not on a cron schedule. Intervals let each subvolume define its own cadence. |
| `Tier` (1-3) controlling schedule | `priority` (1-3) controlling order only | Tiers were too rigid. Per-subvolume policies with inherited defaults give more control with less conceptual overhead. |
| Snapshot naming: `YYYYMMDD-shortname` | `YYYYMMDD-HHMM-shortname` | Sub-daily snapshots need time in the name. Legacy format still recognized for migration. |
| Fixed retention counts per subvolume | Graduated retention (hourly/daily/weekly/monthly windows) + space-governed external retention | Time Machine keeps everything until the disk is full. Space-aware retention protects the 128GB NVMe system drive from filling up while maximizing history on the larger btrfs-pool. |
| `[retention.graduated]` with daily_keep/weekly_keep/monthly_keep | `[defaults.local_retention]` with hourly/daily/weekly/monthly + `min_free_bytes` per root | The hourly window is new (matches frequent snapshotting), and space pressure mode aggressively thins when the filesystem is low. |
| External retention as fixed count | Space-governed + graduated thinning with per-drive `max_usage_percent` and `min_free_bytes` | Fills the drive intelligently instead of wasting space or losing history. |
| `local_schedule` + `external_schedule` per subvolume | `snapshot_interval` + `send_interval` (decoupled) | Snapshot creation and external sends happen on independent intervals. Sends are on-plug + periodic. |
| All config fields required per subvolume | `[defaults]` section with per-subvolume overrides | Most subvolumes share the same policy. Only override what differs. |

### What Stayed the Same

- **Planner/executor separation** — the most important architectural property. `plan.rs` is a pure function producing a `BackupPlan`. It reads config and filesystem state through a trait but never modifies anything.
- **FileSystemState trait** — `MockFileSystemState` for unit tests, `RealFileSystemState` for production. Exactly as planned.
- **Pin file format** — `.last-external-parent-{DRIVE_LABEL}` with legacy fallback. Preserved for coexistence with the bash script during parallel running.
- **PlannedOperation variants** — CreateSnapshot, SendIncremental, SendFull, DeleteSnapshot, PinParent. Same as the plan specified.
- **Module boundaries** — config.rs, types.rs, plan.rs, retention.rs, chain.rs, drives.rs, cli.rs, commands/. The responsibility separation from CLAUDE.md was followed.

## Decisions Made During Implementation

1. **GraduatedRetention uses `Option<u32>` fields** — this allows per-subvolume partial overrides that merge with defaults. A subvolume can override `daily = 7` without having to restate all four fields.

2. **Space pressure is a boolean flag to graduated_retention()** — when the filesystem is below `min_free_bytes`, the hourly window is thinned to 1 per hour instead of keeping everything. This is the first line of defense for the NVMe drive. The executor (Phase 2) will check whether actual space was recovered and can delete more aggressively if needed.

3. **Legacy snapshot names are parsed as midnight** — `20260322-opptak` becomes `2026-03-22 00:00 opptak`. This means they sort correctly alongside new `YYYYMMDD-HHMM-shortname` snapshots and retention logic handles them without special cases.

4. **`send_interval` is decoupled from `snapshot_interval`** — a subvolume can snapshot every 15 minutes but only send to external every hour. This reduces I/O load on external drives while maintaining fine-grained local recovery points.

5. **The planner checks "already on external" before planning sends** — if today's snapshot is already on WD-18TB1, no send is planned. This makes the plan idempotent and safe to run repeatedly.

## What PLAN.md Needs Updated

The plan document still describes the original bash-port design. It should be updated to reflect:

1. Interval-based scheduling replaces cron-like schedules
2. The `[defaults]` config section and inheritance model
3. Space-aware local retention and space-governed external retention
4. The `YYYYMMDD-HHMM-shortname` snapshot naming
5. `priority` replacing `tier` in subvolume config
6. The `send_interval` / `snapshot_interval` decoupling
7. Phase 1 deliverable is now "urd plan matches Time Machine-style planning" rather than "matches bash script"

## Open Questions for Phase 2

- **How should `urd backup` handle the first run?** Legacy snapshots exist on disk but not in SQLite. The `urd init` command needs to import them. Should `urd backup` refuse to run without init, or auto-detect and import?
- **Incremental send parent resolution when the pin points to a legacy-named snapshot** — the parent path uses the legacy name, send uses the new name. This should work since btrfs send/receive doesn't care about directory names, only subvolume UUIDs.
- **systemd timer interval** — the minimum `snapshot_interval` across all subvolumes is 15m (htpc-home). The timer should fire at least that often. Should we generate the timer unit automatically or document the required interval?
